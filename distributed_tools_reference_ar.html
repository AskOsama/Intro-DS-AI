القسم 1

نماذج الحوسبة (Computing Paradigms)

فهم نموذج الحوسبة أساسي لاختيار الأداة الصحيحة. كل نموذج يمثل نهجاً مختلفاً لتنظيم وتنفيذ الحسابات الموزعة.

MapReduce

نموذج الحوسبة الموزعة الأصلي باستخدام نهج ثنائي المراحل صارم: Map (تحويل) ← Shuffle ← Reduce (تجميع). البيانات تتدفق عبر القرص بين المراحل.

  • نموذج تنفيذ بسيط وقابل للتنبؤ
  • عبء I/O عالٍ على القرص
  • الأفضل لأعمال ETL الدفعية
  • مرونة محدودة لخطوط الأنابيب المعقدة
الأدوات: Hadoop

DAG-Based (قائم على الرسم البياني الموجه غير الدوري)

الحسابات منظمة كرسم بياني للعمليات يُحسّن قبل التنفيذ. يسمح بخطوط أنابيب متعددة المراحل معقدة مع فرص تحسين.

  • التقييم الكسول (بناء الرسم ثم التنفيذ)
  • تحسين الاستعلامات ممكن
  • أكثر مرونة من MapReduce
  • يمكن تحسين نقل البيانات
الأدوات: Spark, Dask, Airflow, Prefect

Dataflow / Stream Processing (معالجة التدفق)

البيانات تتدفق باستمرار عبر المشغلين. المعالجة الحقيقية للتدفق تعالج الأحداث واحداً تلو الآخر؛ الدفعات الصغيرة تعالج دفعات صغيرة بشكل متكرر.

  • معالجة منخفضة الكمون
  • دلالات وقت الحدث
  • النوافذ والمحفزات
  • ضمانات exactly-once ممكنة
النوع الفرعيالوصفالأدوات
True Streamingمعالجة حدث بحدثFlink
Micro-batchفترات دفعات صغيرةSpark Streaming
Unified Modelنفس API للدفعي والتدفقBeam, Flink

Task-Based / Actor-Based (التوازي القائم على المهام/الفاعلين)

الحساب مقسم إلى مهام مستقلة أو فاعلين ذوي حالة. أكثر مرونة من مناهج التوازي البياني.

  • توازي دقيق الحبيبات
  • يدعم أعباء العمل غير المتجانسة
  • توليد مهام ديناميكي
  • جيد للحسابات غير المنتظمة
النموذجالوصفالأدوات
Task-basedدوال متوازية بدون حالةRay, Parsl
Actor-basedكائنات موزعة ذات حالةRay
Hybridمهام وفاعلين معاًRay

Dataflow Graphs - ML (الرسوم البيانية لتدفق ML)

الحسابات كرسوم بيانية حيث العقد عمليات والحواف tensors/مصفوفات. محسّنة لأعباء العمل العددية و ML.

  • التفاضل التلقائي
  • تحسين خاص بالعتاد (GPU/TPU)
  • بناء الرسم البياني ثابت أو ديناميكي
  • دمج النواة والتجميع
نوع الرسمالوصفالأدوات
Static (Define-then-run)الرسم يُبنى أولاً ثم يُنفذTensorFlow (graph mode)
Dynamic (Define-by-run)الرسم يُبنى أثناء التنفيذPyTorch, TensorFlow (eager)
Functionalتحويلات قابلة للتركيبJAX

GPU-Accelerated Data Processing (معالجة البيانات المُسرّعة بـ GPU)

تستفيد من توازي GPU لمهام معالجة البيانات التي تُجرى تقليدياً على CPU.

  • توازي ضخم (آلاف الأنوية)
  • تحسين عرض نطاق الذاكرة
  • تنسيقات بيانات عمودية (Arrow)
  • محدود بذاكرة GPU
الأدوات: RAPIDS, RAPIDS+Dask

Collective Communication (الاتصال الجماعي للتدريب الموزع)

يزامن المشتقات/المعاملات عبر العمال باستخدام عمليات جماعية فعالة.

  • Ring-allreduce لكفاءة النطاق الترددي
  • توازي البيانات (النموذج مكرر)
  • توسع شبه خطي
  • غير مرتبط بإطار عمل معين
الأدوات: Horovod, PyTorch Distributed, TensorFlow Distributed

Workflow Orchestration (تنسيق سير العمل)

يدير تبعيات المهام والجدولة، وليس معالجة البيانات نفسها.

  • DAG من المهام (وليس عمليات البيانات)
  • الجدولة والمراقبة
  • يُشغّل أنظمة خارجية
  • إعادة المحاولة والتنبيه
الأدوات: Airflow, Prefect

القسم 3

أنماط المعالجة

المعالجة الدفعية فقط (Batch Processing Only)

معالجة مجموعات بيانات محدودة في مهام مجدولة. كمون أعلى، إنتاجية أعلى.

  • مجموعة البيانات الكاملة متاحة
  • محسّنة للإنتاجية
  • تنفيذ مجدول/مُشغّل
  • النتائج بعد اكتمال المهمة
الأدوات: Hadoop, Dask, Parsl, RAPIDS, XGBoost, Horovod, JAX

معالجة التدفق فقط (Stream Processing Only)

معالجة بيانات غير محدودة باستمرار بكمون منخفض.

  • الأحداث تُعالج عند وصولها
  • كمون منخفض (ميلي ثانية إلى ثوانٍ)
  • استعلامات مستمرة
  • إدارة الحالة مطلوبة
الأدوات: Flink (التصميم الأساسي)

دفعي + تدفق موحد (Unified Batch + Stream)

نفس الواجهة والدلالات لأعباء العمل الدفعية والتدفقية.

النهجالوصفالأدوات
Micro-batch streamingدفعات صغيرة تحاكي التدفقSpark
True unifiedتدفق أولاً مع الدفعي كحالة خاصةFlink, Beam

عام الغرض / تفاعلي (General-Purpose / Interactive)

يدعم الدفعي، الاستعلامات التفاعلية، وأعباء عمل الخدمة.

  • تنفيذ مهام منخفض الكمون
  • تطوير تفاعلي
  • قدرات خدمة النماذج
  • أعباء عمل غير متجانسة
الأدوات: Ray, Spark (تفاعلي)

ملخص أنماط المعالجة

القسم 6

أنماط البنية

بنية السيد-العامل (Master-Worker Architecture)

منسق مركزي (السيد/المشغّل) يوزع العمل على العمال.

  • جدولة مركزية
  • السيد يمكن أن يكون عنق زجاجة
  • نموذج ذهني بسيط
  • نقطة فشل واحدة (تُخفف بـ HA)
الأدوات: Spark (Driver-Executor), Hadoop (JobTracker-TaskTracker), Flink (JobManager-TaskManager), Dask (Scheduler-Worker)

لامركزي / نظير-لنظير (Decentralized / Peer-to-Peer)

لا منسق واحد؛ العمال يتواصلون مباشرة.

  • لا عنق زجاجة واحد
  • تنسيق أكثر تعقيداً
  • إمكانية تسامح أفضل مع الأخطاء
  • طوبولوجيات الحلقة شائعة
الأدوات: Horovod (ring-allreduce)

بنية Parameter Server (خادم المعاملات)

خوادم مخصصة تخزن معاملات النموذج؛ العمال يحسبون المشتقات.

  • تدريب غير متزامن ممكن
  • يتوسع لعمال كثيرين
  • عنق زجاجة الاتصال عند PS
  • مشاكل القِدم ممكنة
الأدوات: TensorFlow (ParameterServerStrategy)

الاتصال الجماعي (Collective Communication)

العمال يتواصلون مباشرة باستخدام عمليات جماعية (all-reduce, broadcast, إلخ).

العمليةالوصف
All-reduceتجميع + توزيع (المشتقات)
Broadcastواحد-للكل (المعاملات)
Scatter/Gatherتوزيع/جمع البيانات
الأدوات: Horovod, PyTorch DDP, TensorFlow MirroredStrategy

النمطحالة الاستخدامالأدوات

هرميكمون منخفض، مهام كثيرةRay

القسم 7

استراتيجيات تقسيم البيانات

التقسيم القائم على الكتل (Block-Based Partitioning)

البيانات تُقسم إلى كتل ثابتة الحجم موزعة عبر العقد. بسيط، قابل للتنبؤ، جيد للوصول المتسلسل.

الأدوات: Hadoop (كتل 128MB), Spark (كتل HDFS)

التقسيم بالتجزئة (Hash Partitioning)

البيانات موزعة بناءً على hash المفتاح. توزيع متساوٍ، تجميع البيانات المتعلقة، جيد لـ joins/aggregations.

الأدوات: Spark, Flink, Dask

تقسيم ذاكرة GPU

البيانات مقسمة لتناسب قيود ذاكرة GPU. محدودة بـ RAM الـ GPU، نموذج partition-per-GPU، عرض نطاق عالي داخل GPU.

الأدوات: RAPIDS, RAPIDS+Dask

تقسيم الصفوف مقابل الأعمدة (ML)

Feature parallelالأعمدة مقسمةXGBoost
Model parallelطبقات النموذج مقسمةPyTorch RPC, TF

القسم 8

آليات التسامح مع الأخطاء

تكرار البيانات (Data Replication)

نسخ متعددة من البيانات عبر العقد. استرداد بسيط، عبء تخزين (2-3x)، عامل تكرار قابل للتكوين.

الأدوات: Hadoop (تكرار HDFS), Dask (اختياري)

الاسترداد القائم على النسب (Lineage-Based Recovery)

تتبع التحويلات؛ إعادة حساب البيانات المفقودة من المصدر. لا عبء تخزين، تكلفة إعادة الحساب، يعمل جيداً مع البيانات غير القابلة للتغيير.

الأدوات: Spark (RDD lineage), Ray (object lineage)

Checkpointing (نقاط التفتيش)

حفظ الحالة دورياً إلى التخزين الدائم.

النوعالوصفالأدوات
دوريلقطات زمنيةFlink, Spark, TensorFlow, PyTorch
متزامنلقطات موزعة متسقةFlink (Chandy-Lamport)
Savepointsيُشغّل بواسطة المستخدم للترقياتFlink

إعادة محاولة على مستوى المهمة (Task-Level Retry)

إعادة محاولة المهام الفاشلة تلقائياً. بسيط وفعال، عدد إعادات قابل للتكوين، تراجع أسي.

الأدوات: جميع أدوات سير العمل (Airflow, Prefect, Parsl), Spark, Dask, Ray

التنفيذ التخميني (Speculative Execution)

تشغيل مهام مكررة للمتأخرين؛ استخدام أول نتيجة. يتعامل مع العقد البطيئة، استخدام موارد إضافي.

الأدوات: Hadoop, Spark

التنفيذ المرن (Elastic Execution)

إضافة/إزالة العمال ديناميكياً أثناء التنفيذ. يتعامل مع فشل العقد بأناقة، يدعم instances الفورية.

الأدوات: Ray, Horovod (Elastic), PyTorch (TorchElastic)

ملخص التسامح مع الأخطاء

الآليةالأدوات
تكرار البياناتHadoop, Dask

CheckpointingFlink, Spark, TF, PyTorch
إعادة محاولة المهمةجميع الأدوات
التنفيذ التخمينيHadoop, Spark
التنفيذ المرنRay, Horovod, PyTorch

القسم 9

نظرة عامة على نقاط القوة والقيود

أبطال التوسع

  • دفعي بيتابايت: Hadoop, Spark
  • وقت حقيقي على نطاق واسع: Flink, Kafka
  • مجموعات GPU: RAPIDS+Dask, TensorFlow, PyTorch
  • حواسيب HPC الفائقة: Parsl

قادة سهولة الاستخدام

  • علماء بيانات Python: Dask (واجهات مألوفة), Ray (decorators بسيطة)
  • ممارسو ML: PyTorch (بديهي), Keras/TensorFlow
  • مؤلفو سير العمل: Prefect (Pythonic), Airflow (ناضج)

قادة الأداء

  • المعالجة في الذاكرة: Spark (10-100x مقابل Hadoop)
  • تسريع GPU: RAPIDS (10-100x مقابل CPU pandas)
  • التدريب الموزع: Horovod, PyTorch DDP (توسع شبه خطي)
  • كمون الوقت الحقيقي: Flink (ميلي ثانية)

أبطال المرونة

  • عام الغرض: Ray (مهام، فاعلين، خدمة، تدريب)
  • متعدد المواقع: Parsl (laptop → HPC → cloud)
  • خطوط أنابيب قابلة للنقل: Beam (اكتب مرة، شغّل في أي مكان)
  • أعباء عمل هجينة: Spark (دفعي، تدفق، SQL، ML، رسم بياني)
